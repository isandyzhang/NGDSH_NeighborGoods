# NeighborGoods

NeighborGoods is a community-focused second-hand marketplace platform for social housing residents.  
The system is split into a React frontend and a .NET backend API, deployed on Azure.

## 1. Architecture

### 1.1 Runtime Components

- Frontend: `NeighborGoods.Frontend` (React + Vite + TypeScript)
- Backend: `NeighborGoods.Api` (ASP.NET Core Minimal API)
- Realtime: SignalR hub at `/hubs/messages`
- Database: Azure SQL
- Object storage: Azure Blob Storage
- Deployment targets:
  - Frontend: Azure Static Web Apps
  - Backend: Azure Container Apps

### 1.2 Repository Layout

```text
NGDSH_NeighborGoods/
├─ NeighborGoods.Frontend/        # React SPA
├─ NeighborGoods.Api/             # ASP.NET Core API
├─ NeighborGoods.Api.Tests/       # Backend tests
├─ infra/bicep/                   # Azure IaC (Bicep modules + parameters)
├─ .github/workflows/             # CI/CD workflows
├─ Dockerfile.api                 # Backend container image definition
└─ NeighborGoods.sln
```

## 2. Tech Stack

### 2.1 Frontend

- React 19
- React Router
- Vite 8
- TypeScript
- Tailwind CSS
- Axios
- `@microsoft/signalr`

### 2.2 Backend

- .NET 9 (ASP.NET Core Minimal API)
- Entity Framework Core (SQL Server provider)
- JWT authentication
- SignalR
- Health checks (DB + Blob)
- Built-in rate limiting

## 3. Core Backend Endpoints

The API is organized by feature modules and mapped in `Program.cs`.

- System / health:
  - `GET /health`
- Authentication:
  - `POST /api/v1/auth/login`
  - `POST /api/v1/auth/refresh`
  - `POST /api/v1/auth/revoke`
  - `GET /api/v1/auth/line/login`
  - `GET /api/v1/auth/line/callback`
- Realtime:
  - `GET /hubs/messages` (SignalR)
- LINE webhook:
  - `POST /api/v1/integrations/line/webhook`

## 4. Local Development

## 4.1 Prerequisites

- .NET SDK 9.x
- Node.js 20+ (22 is also supported)
- SQL Server (or Azure SQL)
- Azure Storage account (or development equivalent for local testing)

### 4.2 Frontend Setup

```bash
cd NeighborGoods.Frontend
npm install
```

Create a local environment file:

```bash
cp .env.example .env
```

Default values in `.env.example`:

```env
VITE_API_BASE_URL=http://localhost:5065
VITE_SIGNALR_BASE_URL=http://localhost:5065
```

Run frontend:

```bash
npm run dev
```

### 4.3 Backend Setup

Set required environment variables (or user secrets) before running backend.

Minimum required settings:

- `ConnectionStrings__DefaultConnection`
- `Jwt__Issuer`
- `Jwt__Audience`
- `Jwt__SigningKey` (>= 32 chars)
- `AzureBlob__ConnectionString`
- `AzureBlob__ContainerName`

LINE login settings:

- `Line__ChannelId`
- `Line__ChannelSecret`
- `Line__CallbackUrl`
- `Line__Scope` (default: `openid profile`)

Run backend:

```bash
cd NeighborGoods.Api
dotnet restore
dotnet run
```

Default development URL from launch profile:

- `http://localhost:5065`
- `https://localhost:7233`

## 5. Production Configuration

### 5.1 Frontend

Production frontend must point to public API domain:

- `VITE_API_BASE_URL=https://api.neighborgoodstw.com`
- `VITE_SIGNALR_BASE_URL=https://api.neighborgoodstw.com`

For SPA refresh/deep-link support on Azure Static Web Apps, keep:

- `NeighborGoods.Frontend/public/staticwebapp.config.json`
  - `navigationFallback.rewrite = /index.html`

### 5.2 Backend

Container App environment variables are generated by Bicep from:

- `infra/bicep/deploy.parameters.json`
- `infra/bicep/modules/containerapp.bicep`

Production values should not use localhost for:

- `Line__CallbackUrl`
- `Cors__AllowedOrigins__0`
- `Cors__AllowedOrigins__1`
- `LineMessagingApi__BaseUrl`

## 6. CI/CD

Current workflows are under `.github/workflows/`.

- `backend_csharp_docker.yml`
  - Builds backend Docker image
  - Pushes to GHCR
  - Updates Azure Container App image
- `frontend_react.yml`
  - Builds frontend artifact
- `infra_bicep.yml`
  - Deploys Azure infrastructure from Bicep templates

If you want full frontend auto-deploy, ensure the workflow includes Static Web Apps deployment step with deployment token.

## 7. Infrastructure as Code

Main files:

- `infra/bicep/deploy.bicep`
- `infra/bicep/deploy.parameters.json`
- `infra/bicep/modules/*.bicep`

Deploy via Azure CLI:

```bash
az deployment group create \
  --resource-group <resource-group> \
  --template-file infra/bicep/deploy.bicep \
  --parameters infra/bicep/deploy.parameters.json
```

## 8. LINE Integration Notes

LINE Login callback URL should be frontend route:

- `https://www.neighborgoodstw.com/auth/line/callback`

LINE Messaging API webhook URL should be backend endpoint:

- `https://api.neighborgoodstw.com/api/v1/integrations/line/webhook`

Do not mix callback and webhook URLs.

## 9. Security Requirements

- Do not commit production secrets in repository files.
- Move sensitive values to:
  - Azure Key Vault, or
  - GitHub Secrets + environment variables
- Rotate any credentials that were previously committed.

Sensitive items include:

- SQL connection strings
- Storage account keys
- JWT signing keys
- LINE channel secrets and tokens

## 10. Validation Checklist

After each deployment:

1. Frontend
   - `https://www.neighborgoodstw.com/` returns app UI
   - Direct access and refresh on `/listings` works
2. Backend
   - `https://api.neighborgoodstw.com/health` returns healthy
3. LINE OAuth
   - `/api/v1/auth/line/login` redirects with production `redirect_uri`
